# Task 5 Report: Drift v3 Photo Nutrition Persistence

## Outcome

Implemented confirmed photo-estimate persistence behind
`NutritionRepository.logPhotoEstimate`:

- one `PhotoNutritionLog` input carries only the confirmed name, meal time,
  image type, deterministic nutrition ranges, confidence, and calculation
  summary;
- one Drift transaction inserts the logged-food row and updates
  `daily_nutritions`;
- existing integer columns and daily totals use the same rounded midpoint
  values;
- range, confidence, image-type, source, and summary metadata remain attached
  to the logged row;
- deleting the row subtracts those midpoint values through the existing delete
  path;
- manual `logFood` behavior remains unchanged and receives the schema default
  `MANUAL` source with null analysis metadata.

No analysis ID, image bytes, provider output, uncertainty payload, or
correction data has a persistence column or write path.

## Schema and migration

- Increased `GymDatabase.schemaVersion` from 2 to 3.
- Added `source` with default `MANUAL`.
- Added nullable calorie/macro range, confidence, image-type, and calculation
  summary columns.
- The `from < 2` migration block remains unchanged.
- The `from < 3` block adds each new column through Drift's migrator.
- A file-backed v2 fixture verifies that an old manual row survives with its
  original name/calories, `source = MANUAL`, and null photo metadata.
- `database.g.dart` was regenerated by build_runner. No generated file was
  hand-edited, and `logged_food_dao.g.dart` did not change.

`--delete-conflicting-outputs` is obsolete and ignored, then generated the
expected Drift output.

## Validation and rounding

- Name is trimmed, required, and capped at the canonical 160 characters.
- Calculation summary is trimmed and required.
- Unknown image types are rejected.
- Every range is revalidated at the persistence boundary for finite,
  non-negative, ordered values before the transaction opens.
- Persistence bounds are 10,000 kcal and 1,000 g per macro.
- The documented rounding helper uses Dart nearest-integer rounding for the
  non-negative midpoint values written to both the row and daily totals.

The approved plan requires physiological bounds but does not assign numeric
values. This implementation makes the boundary explicit at 10,000 kcal and
1,000 g per macro. Those constants are isolated in the repository and can be
adjusted after product review without a schema migration.
